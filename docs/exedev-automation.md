# exe.dev Instance Automation

## Summary

exe.dev provides an official HTTPS API, but it is not a traditional REST
resource API. It is a command API with one endpoint:

```sh
POST https://exe.dev/exec
```

The request body is the exe.dev command to run, with the same semantics as:

```sh
ssh exe.dev <command>
```

API responses enable JSON output by default, equivalent to using `--json` with
the CLI.

There are two equivalent automation entrypoints:

1. Run `ssh exe.dev <command>` from local scripts or CI.
2. Call `POST https://exe.dev/exec` from a service or native client.

Recommended approach:

1. Use `ssh exe.dev ... --json` for local scripts because it is easy to debug.
2. Use `POST /exec` for long-running automation services that should not depend
   on interactive SSH.
3. Treat `fleet.yaml` as the desired state.
4. Compare the desired VM list with the current VM list from exe.dev.
5. Run `new` for missing VMs.
6. Mark extra VMs and require explicit confirmation before running `rm`.
7. After VM creation, hand nodes to Ansible for Tailscale, k3s, and node labels.

## HTTPS API Usage

Minimal request:

```sh
curl -X POST https://exe.dev/exec \
  -H "Authorization: Bearer $EXE_DEV_API_KEY" \
  -d 'ls'
```

Create a VM:

```sh
curl -X POST https://exe.dev/exec \
  -H "Authorization: Bearer $EXE_DEV_API_KEY" \
  -d 'new --name=p1-a-1 --image=ubuntu:22.04 --no-email'
```

Set the proxy port:

```sh
curl -X POST https://exe.dev/exec \
  -H "Authorization: Bearer $EXE_DEV_API_KEY" \
  -d 'share port p1-a-1 8080'
```

Important limits:

- `/exec` accepts only `POST`.
- There is no stdin.
- There is no pty.
- Request body size is limited to 64KB.
- A single command times out after 30 seconds.
- Failed commands return `422`.
- Token permission failures return `403`.
- Rate limits are applied per SSH key.

## API Token

exe.dev tokens are generated by signing permissions locally with an SSH private
key. For automation, use a dedicated SSH key so it can be revoked independently.

Create a dedicated key:

```sh
ssh-keygen -t ed25519 -C api -f ~/.ssh/exe_dev_api
```

Add the public key to exe.dev:

```sh
cat ~/.ssh/exe_dev_api.pub | ssh exe.dev ssh-key add
```

Core token generation flow:

```sh
b64url() { tr -d '\n=' | tr '+/' '-_'; }

export PERMISSIONS='{"cmds":["ls","new","whoami","share show","share port","share set-public","share set-private"],"exp":1798761600}'
export PAYLOAD=$(printf '%s' "$PERMISSIONS" | base64 | b64url)
export SIG=$(printf '%s' "$PERMISSIONS" | ssh-keygen -Y sign -f ~/.ssh/exe_dev_api -n v0@exe.dev)
export SIGBLOB=$(echo "$SIG" | sed '1d;$d' | b64url)
export EXE_DEV_API_KEY="exe0.$PAYLOAD.$SIGBLOB"
```

`PERMISSIONS` is embedded in the token as plaintext JSON. Do not put secrets in
it.

Common permission fields:

- `exp`: expiration time as a UTC Unix timestamp. Always set it when possible.
- `nbf`: not-before time as a UTC Unix timestamp.
- `cmds`: list of exe.dev commands this token may run.
- `ctx`: uninterpreted context for exe.dev, passable to VM HTTP services.

Default `cmds` include `help`, `ls`, `new`, `whoami`, `ssh-key list`,
`share show`, and `exe0-to-exe1`. Commands such as `rm`, `rename`, `restart`,
and `share port` must be added explicitly.

## Common Commands

List VMs:

```sh
ssh exe.dev ls --json
```

Create a VM:

```sh
ssh exe.dev new --name=p1-a-1 --image=ubuntu:22.04 --no-email --json
```

Create a VM with environment variables:

```sh
ssh exe.dev new --name=p1-a-1 --env FOO=bar --env BAZ=qux --json
```

Delete a VM:

```sh
ssh exe.dev rm p1-a-1
```

Set the HTTP proxy port:

```sh
ssh exe.dev share port p1-a-1 8080 --json
```

Make the HTTP proxy public:

```sh
ssh exe.dev share set-public p1-a-1 --json
```

Restore private access:

```sh
ssh exe.dev share set-private p1-a-1 --json
```

## Recommended Control Flow

```text
fleet.yaml
  -> render desired VM list
  -> ssh exe.dev ls --json to fetch current VM list
  -> diff
  -> create missing VMs
  -> write Ansible inventory
  -> initialize nodes with Ansible
  -> kubectl label/taint nodes
  -> deploy apps with Flux CD or Argo CD
```

## Kubernetes Boundary

exe.dev manages VM lifecycle.

Kubernetes manages workload scheduling.

Do not let Kubernetes create exe.dev VMs directly. You could build a controller
that calls `POST /exec`, but the cleaner boundary is:

```text
external scripts/Ansible: manage VMs
Kubernetes: manage Pods
Flux CD or Argo CD: manage application declarations
```

If autoscaling is needed later, start with an independent reconciler:

```text
fleet.yaml -> desired VMs
POST /exec "ls" -> current VMs
diff -> POST /exec "new ..." / confirmed "rm ..."
```

Do not start with a Kubernetes Cluster Autoscaler provider. exe.dev resources
are shared quota, so the platform fits declarative batch management better than
a cloud-provider model that assumes fixed dedicated CPU/RAM for each VM.
